# OneStep Control Plane 项目说明书（产品 + UI 重设计基线）

> 更新日期：2026-03-19
> 适用对象：产品、设计、前端、后端、运维、演示讲解人员

## 1. 项目是什么

`OneStep Control Plane` 是一个面向 `OneStep runtime` 的控制平面。它不是通用监控大盘，也不是单纯的日志页，而是把运行时实例的接入、状态监控、任务健康、控制命令和操作审计放到一个统一控制台里。

当前仓库是一个 monorepo，主要包含两部分：

- `backend`：负责 Agent WebSocket 接入、状态持久化、查询 API、命令下发与命令生命周期管理。
- `frontend`：负责浏览器控制台，提供服务、任务、实例等多层级视图，以及任务级 / 实例级控制操作。

从代码和现有文档来看，这个系统的核心目标有两类：

1. 让内部运维/研发能够快速判断某个服务是否健康、哪些任务异常、哪些实例离线，以及是否可以安全地下发控制命令。
2. 让团队能以较为清晰的方式演示 `OneStep runtime` 已经具备“可观测 + 可控制 + 可审计”的能力。

## 2. 系统在解决什么问题

传统的运行时管理，往往把“采集、监控、控制、审计”分散在多套工具中。当前项目试图把它们压缩到一个控制台里：

- Agent 通过 WebSocket 主动连入控制平面。
- 控制平面记录实例、任务、事件、指标窗口、会话、命令。
- 浏览器用户从服务视角进入，再逐步下钻到任务和实例。
- 用户可以在控制台查看服务健康，也可以对实例或任务执行操作。
- 所有控制动作以命令记录的形式留痕，便于回溯。

这意味着该产品的价值不只是“展示状态”，而是“在知道发生了什么的同时，能做出有限且受控的动作”。

## 3. 核心用户与使用场景

### 3.1 主要用户

- 内部运维 / SRE：关注服务是否在线、实例是否断连、拓扑是否漂移、任务是否失败。
- 研发 / 平台工程师：关注任务定义、运行时能力、会话状态、命令能力协商结果。
- 产品 / 销售 / 管理层演示人员：需要快速解释系统能看到什么、能控制什么、如何保证操作安全。

### 3.2 核心使用场景

#### 场景 A：服务级巡检

用户进入服务列表，筛选环境后查看：

- 哪些服务在线
- 某服务有多少实例在线 / 离线
- 最近一次同步时间
- 当前部署版本
- 拓扑是否一致

#### 场景 B：问题定位

用户进入服务详情页，查看：

- 失败任务数
- 实例在线情况
- 健康分布
- 是否存在拓扑漂移
- 哪些任务需要优先关注

随后下钻到任务详情或实例详情继续排查。

#### 场景 C：任务控制

用户进入任务详情，对某个任务相关实例执行：

- `pause_task`
- `resume_task`
- `discard_dead_letters`
- `replay_dead_letters`

这类操作偏“运维恢复”而不是“业务配置”。

#### 场景 D：实例控制

用户进入实例详情，对单个实例执行：

- `ping`
- `sync_now`
- `flush_metrics`
- `flush_events`
- `drain`
- `restart`
- `shutdown`

其中部分命令受 runtime 能力协商结果约束，部分命令要求填写 reason。

## 4. 系统结构概览

### 4.1 运行链路

系统可以概括为 4 个层次：

1. `OneStep runtime / Agent`
   负责上报运行时状态、任务指标、任务事件，并与控制平面建立 WS 会话。
2. `Control Plane Backend`
   负责接收 Agent 数据、持久化、提供查询 API、创建命令记录、管理命令生命周期。
3. `Database`
   负责保存服务、实例、任务定义、任务事件、指标窗口、Agent 会话、Agent 命令等。
4. `Control Console Frontend`
   负责把这些数据组织成服务视图、任务视图、实例视图，并承载控制动作。

### 4.2 技术结构

- 后端：FastAPI + SQLAlchemy + Alembic
- 前端：Vite + React + TypeScript + React Router + TanStack Query
- 接入方式：Agent 使用 `GET /api/v1/agents/ws`
- 控制台认证：支持共享用户名/密码登录，登录后通过 session cookie 访问查询接口
- 部署方式：支持本地开发、Docker Compose、镜像部署

### 4.3 当前控制模型

当前控制面并不是任意执行平台，而是“命令型控制台”：

- 每个控制动作都会落成一条或多条 `agent_commands`
- 命令会经历 `pending -> dispatched/accepted/succeeded/failed/...` 等状态
- 命令执行对象是实例，任务级控制本质上也是对绑定实例做 fanout
- 某些命令可以排队等待实例重连后再投递
- 控制能力取决于 Agent 会话协商出来的 `accepted_capabilities`

## 5. 核心领域对象

为了做 UI 重设计，必须先明确页面到底在围绕哪些对象工作。

### 5.1 Service

`Service` 是业务上最顶层的运行单元，按 `(name, environment)` 唯一标识。

它代表：

- 一个逻辑服务
- 某个环境下的一组实例
- 一组任务定义
- 一个服务级健康观察入口

界面上，服务是用户最常见的起点。

### 5.2 Instance

`Instance` 是运行时实例，具有：

- `instance_id`
- `node_name`
- `hostname`
- `pid`
- 部署版本 / OneStep 版本 / Python 版本
- 最近同步时间 / 最近看到时间
- 连接状态 `online/offline/never_reported`
- 健康状态 `ok/degraded/error/starting/unknown`

实例是控制动作的最小执行单位。

### 5.3 Task

`Task` 是服务内的任务定义与运行摘要，关联：

- 任务描述
- source 信息
- emit connector
- 并发度
- 超时
- retry policy
- 拓扑 hash
- 指标窗口统计
- 最近事件分布

任务页承担“任务健康 + 恢复操作 + 运行绑定关系”的职责。

### 5.4 AgentSession

`AgentSession` 表示某个实例与控制平面的会话状态，包含：

- `session_id`
- `status`
- `protocol_version`
- `capabilities`
- `accepted_capabilities`
- 连接时间、最近消息时间、断开时间

对 UI 来说，这个对象非常关键，因为它决定：

- 当前实例能不能控制
- 能控制哪些动作
- 是立即投递还是只能排队

### 5.5 AgentCommand

`AgentCommand` 是所有控制动作的审计和执行载体，包含：

- `kind`
- `args`
- `timeout_s`
- `status`
- `ack_status`
- `result`
- `created_by`
- `reason`
- `source_surface`

这个对象让系统具备“可审计、可回放、可解释”的控制面特征。

## 6. 当前信息架构

前端现有路由结构相对清晰，入口关系如下：

```text
Login
  -> Services List
    -> Service Detail
      -> Task Detail
      -> Instance Detail
```

对应页面职责如下。

### 6.1 登录页

作用：

- 处理控制台认证
- 将用户导向原始目标页面

特点：

- 页面简单，更多是权限入口，不承担产品理解任务
- 目前偏基础表单，不具备品牌化或系统说明能力

### 6.2 服务列表页

作用：

- 作为主入口展示服务目录
- 按环境筛选服务
- 搜索服务
- 以服务为单位展示在线情况、部署版本、最近同步时间、拓扑摘要

用户在这里主要回答两个问题：

1. 我要看哪个服务？
2. 当前环境里哪些服务值得先关注？

### 6.3 服务详情页

作用：

- 展示服务级概览
- 展示任务概况与任务列表
- 展示服务上下文
- 展示健康分布
- 展示拓扑漂移信息

当前它既像概览页，也像运营驾驶舱的雏形，但模块较多、信息分散，视觉层级还不足以支撑“快速判断”。

### 6.4 任务详情页

作用：

- 展示任务成功/失败/重试/死信等摘要
- 查看近期任务事件与指标窗口
- 对绑定实例执行任务控制命令
- 查看与该任务相关的实例、会话、命令

这个页面已经具备明显的“操作页”属性，不再只是查看页。

### 6.5 实例详情页

作用：

- 查看实例运行时快照
- 查看 app snapshot
- 查看控制平面会话与能力
- 执行实例级命令
- 查看该实例命令历史

这是当前系统最接近“控制台核心操作区”的页面。

## 7. 关键用户流程

### 7.1 巡检流程

`服务列表 -> 服务详情 -> 任务/实例详情`

目标是从整体到局部快速缩小范围。

### 7.2 故障排查流程

`服务详情 -> 关注异常任务/异常实例 -> 查看事件/会话/命令 -> 决定是否操作`

目标是让用户在操作前已经知道上下文。

### 7.3 控制执行流程

`实例详情 / 任务详情 -> 选择动作 -> 检查能力与投递模式 -> 填写 reason（如需要） -> 下发命令 -> 查看命令结果`

目标是让操作可解释、可追踪，而不是“点了一个按钮，不知道发生了什么”。

## 8. 当前产品优势

从设计重构角度看，这个项目已经具备一些非常好的基础，不需要从零开始构造产品逻辑。

### 8.1 对象模型清晰

服务、实例、任务、会话、命令这 5 个对象关系明确，足以支撑稳定的信息架构。

### 8.2 控制与观测已经打通

很多内部控制台只能看不能动，很多运维页面又只能动不能解释。当前项目已经把“看”和“动”放在同一个模型内。

### 8.3 审计意识较强

命令记录带有状态、来源、原因和执行结果，天然适合设计成可信赖的控制体验。

### 8.4 前端结构具备重构基础

前端已经是 feature-oriented 结构，页面和查询能力边界比较清楚，适合分批做 UI 重设计，不需要推翻技术底座。

## 9. 当前 UI 的主要问题

以下不是代码问题，而是从产品表达和交互层面观察到的主要 UI 问题。

### 9.1 服务详情页承担过多职责

它现在同时负责：

- 服务总览
- 任务列表
- 服务上下文
- 健康分布
- 拓扑漂移展示

这些内容都合理，但还没有被组织成一个强主线。用户可以看到很多信息，却不一定能快速回答“现在最重要的是什么”。

### 9.2 控制能力表达仍然偏技术内部化

例如能力协商、queue mode、accepted capabilities 等都是真实系统能力，但现在的表达更接近工程实现视角，而不是操作决策视角。

用户真正想知道的是：

- 我现在能不能操作？
- 为什么不能？
- 这个操作会马上执行还是排队？
- 风险等级是什么？

### 9.3 任务详情页的信息组织偏“开发友好”

当前任务页包含控制、事件、详情等内容，但更像若干数据板块的集合。对设计重构来说，它需要更强的“任务状态 -> 风险 -> 可恢复动作 -> 影响实例”顺序。

### 9.4 服务列表页还不是强 fleet overview

它已经能筛选和搜索，但还没有真正成为“值班入口”或“演示入口”。目前更像一份美化后的服务目录。

### 9.5 登录页和全局导航壳缺少品牌与定位表达

当前登录页足够可用，但没有帮助首次进入的人迅速理解：

- 这是什么系统
- 它管什么
- 为什么值得信任

同样，侧边导航虽然完整，但对“演示场景”支持有限。

## 10. UI 重设计目标

这次重设计建议不是纯换皮，而是“结构 + 视觉一起升级”。

### 10.1 内部使用目标

- 让值班和排障更快
- 让操作更可控
- 让风险提示更直接
- 让服务、任务、实例三层级的切换更顺手

### 10.2 对外展示目标

- 让系统能力一眼可讲清
- 强化“运行时控制平面”的系统感
- 让核心指标、核心对象、核心操作更具叙事性

### 10.3 统一目标

- 把“状态观察”和“控制执行”明确地组织成一条可信的操作链路

## 11. 重设计优先级建议

### 11.1 第一优先级：服务详情页

原因：

- 它是最核心的中枢页面
- 承担从“总览”进入“决策”的过渡
- 最能体现产品是否像一个真正的 control plane

建议改造方向：

- 建立单一主视图，突出“当前服务态势”
- 把任务、实例、拓扑、健康、最近控制活动组织成主次分明的版块
- 先回答“是否健康、是否一致、是否需要操作”，再展开细节

### 11.2 第二优先级：实例详情页

原因：

- 这是风险最高、也最体现产品可信度的操作页面

建议改造方向：

- 把“状态摘要、会话能力、操作区、命令历史”改成清晰的决策顺序
- 强化能力不足、会话缺失、仅可排队等限制状态
- 对高风险命令形成更强的视觉分级

### 11.3 第三优先级：任务详情页

原因：

- 它承载恢复动作，但当前结构不够突出“处置逻辑”

建议改造方向：

- 把任务页重组为“任务健康概况 + 最近异常 + 恢复操作 + 影响实例”
- 降低原始数据块的分散感

### 11.4 第四优先级：服务列表页

原因：

- 这是所有流程入口
- 也是未来最适合兼顾内部巡检与外部演示的页面

建议改造方向：

- 从“目录页”升级为“fleet overview”
- 更突出异常服务、环境态势、近期波动和版本分布

### 11.5 第五优先级：登录页与全局导航

原因：

- 需要统一品牌化表达，但不应优先于核心业务页

建议改造方向：

- 登录页加入简短系统定位
- 侧边导航强化当前环境、当前对象、最近上下文

## 12. 设计约束与注意事项

重设计时需要保留以下真实系统约束，避免做出“看起来好看但不符合系统能力”的界面。

### 12.1 控制能力不是静态的

命令是否可用，取决于 Agent 会话协商出来的 `accepted_capabilities`。UI 必须忠实表达这一点。

### 12.2 并非所有命令都能排队

当前只有部分命令支持 `queue_until_reconnect`。设计上不能把所有操作都表现为“稍后执行”。

### 12.3 任务控制本质上是实例 fanout

任务级控制不是抽象黑盒，背后仍然要落到一组实例上，因此 UI 需要让“作用范围”清楚可见。

### 12.4 命令必须可审计

高风险或恢复类操作需要 reason，且结果应可追踪。设计上不能把它做成轻飘飘的一次性按钮。

### 12.5 这是控制台，不是营销官网

即使面向演示，也不能牺牲真实信息密度和可操作性。

## 13. 建议的下一步产出

基于这份说明书，下一步建议直接进入页面级重设计方案，而不是继续补概念文档。

推荐输出顺序：

1. `Service Detail` 重设计说明
2. `Instance Detail` 重设计说明
3. `Task Detail` 重设计说明
4. `Services List` 重设计说明
5. 全局视觉语言与组件规范

每个页面的重设计说明建议至少包含：

- 页面目标
- 主要用户问题
- 信息架构
- 核心模块
- 主操作与风险提示
- 桌面端 / 移动端布局策略

## 14. 一句话总结

`OneStep Control Plane` 当前已经具备一个真实控制平面的核心骨架：它能接 runtime、存状态、看任务、看实例、下命令、留审计。接下来最值得做的，不是再加一层“看起来像监控台”的皮，而是把它重构成一个更清晰、更可信、更适合运维和演示双场景的控制台产品。
